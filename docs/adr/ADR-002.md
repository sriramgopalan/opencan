# ADR-002: All Errors Use a Typed AppError Class

## Status
Accepted

## Date
2026-06-13

## Context

JavaScript's `throw` accepts any value. In practice this means errors in a
codebase without a convention are a mix of `new Error('user not found')`,
`throw 'FORBIDDEN'`, `throw { status: 403 }`, and `throw null`. Callers cannot
reliably distinguish error types at runtime without `instanceof` chains against
multiple possible shapes.

In OpenCan specifically, all procedures go through tRPC, which catches thrown
values and serialises them into tRPC error responses. Without typed errors:

- The tRPC error handler cannot distinguish a business logic error (user not
  found, quota exceeded) from an infrastructure failure (database connection
  lost), and cannot map them to appropriate HTTP status codes or tRPC error
  codes.
- Error messages from `new Error(prismaError.message)` leak implementation
  details — table names, constraint names, query parameters — to the client.
- Client code cannot branch on error type without string-matching the `message`
  field, which breaks silently when messages change.
- Logging is inconsistent: some throws carry context objects, most carry only
  a string.

The application handles customer feedback data under GDPR. Leaking database
internals in error messages is a compliance risk as well as a security one.

## Decision

All thrown errors in application code (`/src/`, `/server/`) must be instances
of `AppError`, defined in `/lib/errors.ts`.

`AppError` has the following shape:

```ts
export class AppError extends Error {
  constructor(
    public readonly code: AppErrorCode,
    message: string,
    public readonly meta?: Record<string, unknown>,
  ) {
    super(message);
    this.name = 'AppError';
  }
}

export type AppErrorCode =
  | 'NOT_FOUND'
  | 'FORBIDDEN'
  | 'UNAUTHORIZED'
  | 'VALIDATION_ERROR'
  | 'CONFLICT'
  | 'RATE_LIMITED'
  | 'INTERNAL_ERROR';
```

`AppErrorCode` is a string union defined in the same file. New codes are added
to the union as new error conditions are identified. The union is the exhaustive
list of error conditions the application can signal to callers.

The tRPC error formatter (in `/server/trpc.ts`) catches `AppError` instances
and maps them to tRPC error codes and HTTP status codes. All other thrown values
are caught and re-thrown as `AppError('INTERNAL_ERROR', 'An unexpected error
occurred')` so that raw `Error` instances never reach the client.

**`throw new Error(...)` is forbidden in `/src/` and `/server/`.** ESLint's
`no-restricted-syntax` rule enforces this.

Infrastructure code outside the application boundary (scripts, migrations,
build tools) may use plain `Error`.

## Consequences

**For development:**
- Every error condition has a named code. Adding a new condition means adding
  to the `AppErrorCode` union, which makes the full set of application errors
  visible in one place.
- Client code switches on `error.data.code` (typed by tRPC) rather than
  pattern-matching strings. TypeScript reports unhandled cases when the union
  grows.
- The tRPC error formatter is the single point where errors are sanitised before
  leaving the server. It logs the real error (including `meta`) via Pino and
  returns only the code and a safe message to the client.
- Pino log lines for errors always include `{ code, meta }` in addition to the
  message, making errors filterable and traceable in production.

**For testing:**
- `expect(result).rejects.toMatchObject({ code: 'FORBIDDEN' })` is a stable
  assertion. It does not break when the message text changes.
- The tRPC error formatter can be tested in isolation with a mock that throws
  each `AppErrorCode` and verifies the HTTP status code mapping.

**Trade-offs:**
- Every throw site requires importing `AppError` and choosing a code. This is
  a small overhead that pays for itself at the first client error-handling
  branch.
- The `AppErrorCode` union must be updated when new error conditions are
  introduced. Missing a code causes a TypeScript error, which is the intended
  enforcement mechanism.

## Alternatives Considered

**Result types (`neverthrow`, `ts-results`).**
Rejected. Result types require every function in the call chain to handle or
propagate the `Err` case explicitly. This is valuable in pure functional
codebases but adds significant ceremony to async tRPC procedures where the
exception model is already well-understood. tRPC's error boundary is a cleaner
fit than monadic propagation for this stack.

**HTTP status codes as the error taxonomy.**
Rejected. HTTP codes are a transport concern. `403 Forbidden` and `404 Not
Found` are meaningful over HTTP but not as business logic signals within the
application. `AppErrorCode` values like `FORBIDDEN` and `NOT_FOUND` map to HTTP
codes in the formatter; the application layer does not need to know about HTTP.

**Per-domain error subclasses (`BoardNotFoundError extends Error`).**
Rejected. Subclasses proliferate quickly and client code ends up doing
`instanceof` chains that break across module boundaries. A typed string union
for `code` is simpler, exhaustively checkable by TypeScript, and serialises
cleanly over the tRPC wire format.

**Plain `Error` with a `code` property duck-typed.**
Rejected. Without `instanceof AppError`, the tRPC error formatter cannot
reliably distinguish application errors from framework errors or third-party
library errors that happen to have a `code` property.
