# ADR-006: Structured JSON Logging via Pino

## Status
Accepted

## Date
2026-06-13

## Context

`console.log` is the path of least resistance for logging in Node.js. It is
synchronous, outputs unstructured text, and has no concept of log levels,
context propagation, or field filtering. In a production application handling
customer data, `console.log` produces logs that are:

- **Unsearchable**: a plain text line `"User created: alice@example.com"` cannot
  be queried by field in a log aggregation system (Datadog, Grafana Loki, AWS
  CloudWatch Insights). Finding all log lines related to a specific user requires
  full-text search, which is slow and imprecise.
- **Ungated**: `console.log` has no log level. Debug output written during
  development ships to production alongside important operational events, with
  no way to filter one from the other without removing the calls.
- **A PII risk**: `console.log(user)` or `console.log(req.body)` dumps an entire
  object, including email addresses, names, and other fields that must not appear
  in logs under GDPR (see ADR privacy requirements). There is no mechanism to
  redact fields automatically.
- **Synchronous**: `console.log` blocks the event loop on every write. In a
  high-throughput Node.js server, synchronous I/O for logging measurably reduces
  request throughput.

Etash handles customer feedback data on behalf of its users. Log lines are
processed by a log aggregation pipeline. Log lines that contain PII are a
compliance incident.

## Decision

All logging in application code (`/src/`, `/server/`, `/lib/`) must use the
Pino logger instance exported from `/lib/logger.ts`.

**`console.log`, `console.error`, `console.warn`, and `console.info` are
forbidden in application code.** ESLint's `no-console` rule is enabled for all
files under `/src/`, `/server/`, and `/lib/`. Violations fail CI.

**Logger configuration (`/lib/logger.ts`):**

```ts
import pino from 'pino';

export const logger = pino({
  level: process.env.LOG_LEVEL ?? (process.env.NODE_ENV === 'production' ? 'info' : 'debug'),
  redact: {
    paths: ['*.password', '*.token', '*.secret', '*.apiKey', '*.email', '*.creditCard'],
    censor: '[REDACTED]',
  },
  ...(process.env.NODE_ENV !== 'production' && {
    transport: { target: 'pino-pretty', options: { colorize: true } },
  }),
});
```

**Logging conventions:**

```ts
// Correct — structured context object first, message string second
logger.info({ userId, boardId }, 'board created');
logger.error({ err, userId }, 'board creation failed');
logger.warn({ flagName, userId }, 'feature flag evaluated for inactive user');

// Forbidden — message contains interpolated values
logger.info(`Board ${boardId} created by ${userId}`);

// Forbidden — logging a full object that may contain PII
logger.info({ user }, 'user loaded');

// Correct — destructure to include only safe fields
const { id, createdAt } = user;
logger.info({ userId: id, createdAt }, 'user loaded');
```

**Log levels:**

| Level | When to use |
|-------|-------------|
| `error` | An operation failed and requires attention. Always include `{ err }`. |
| `warn` | An unexpected condition that did not cause failure but may indicate a problem. |
| `info` | A significant application event (resource created, user authenticated, job completed). |
| `debug` | Detailed diagnostic information. Emitted in development only; suppressed in production by default. |

`debug` log lines may include richer context than `info` lines. They must still
not contain raw PII objects — the `LOG_LEVEL=debug` environment variable enables
them in production for incident investigation, and the redact configuration
applies at all levels.

**Child loggers** for request-scoped context:

```ts
// In tRPC context factory
const requestLogger = logger.child({ requestId: crypto.randomUUID() });
```

The child logger propagates `requestId` to every log line emitted during the
request without the caller needing to pass it explicitly.

## Consequences

**For development:**
- Every log line is a JSON object in production. Log aggregation queries like
  `level = "error" AND userId = "usr_123"` work without full-text search.
- The `redact` configuration in `/lib/logger.ts` is the single point where PII
  fields are stripped. A new sensitive field is protected by adding its path to
  the redact list once, not by auditing every log call site.
- `pino-pretty` is used in development for human-readable output. The
  `NODE_ENV` check ensures it is never loaded in production (it is a dev
  dependency and not available in the production bundle).
- Log level is controlled by `LOG_LEVEL` environment variable, validated at
  startup. Changing verbosity does not require a code change or redeploy.

**For compliance:**
- Pino's `redact` option removes PII fields before serialisation. A log call
  that accidentally includes `{ user }` will have `user.email` replaced with
  `[REDACTED]` in the output. This is a defence-in-depth measure — the primary
  rule is never to log PII objects, but the redact configuration catches
  accidental inclusions.
- Log lines contain `userId` (an opaque identifier) rather than `email` (a
  direct identifier). A log line that contains only `userId` is not directly
  identifying under GDPR guidance for pseudonymous identifiers.

**For performance:**
- Pino is the fastest JSON logger in the Node.js ecosystem by a significant
  margin. It serialises log objects asynchronously using a worker thread,
  removing log I/O from the request event loop.

**Trade-offs:**
- `pino-pretty` must be installed as a dev dependency and is not available in
  production. Log output in production is raw JSON, which is unreadable without
  a log viewer. This is the correct trade-off: production logs are read by
  machines first.
- Structured logging requires discipline at the call site. `logger.info('board
  created for ' + userId)` is syntactically valid but defeats the purpose. Code
  review and the ESLint `no-restricted-syntax` rule for string concatenation in
  log calls are the enforcement mechanisms.

## Alternatives Considered

**`console.log` with a structured wrapper.**
Rejected. A wrapper that calls `console.log(JSON.stringify(obj))` is
synchronous, lacks log levels, lacks the redact feature, and lacks child logger
support. It is a worse Pino with more maintenance overhead.

**Winston.**
Rejected. Winston is a capable logger but significantly slower than Pino
(3–5× in benchmarks), has a more complex configuration API, and does not
provide built-in async transport. Pino's performance advantage is meaningful
for a high-throughput feedback ingestion endpoint.

**OpenTelemetry logging.**
Rejected as the primary logger at this stage. OpenTelemetry's logging API is
still maturing and introduces significant additional dependencies. Pino can emit
to an OpenTelemetry collector via a transport if distributed tracing is adopted
in the future. Adopting OTel logging now would be premature.

**Disabling `console.log` via runtime override (`console.log = () => {}`).**
Rejected. Silent suppression makes debugging harder without making logging
better. The ESLint rule that forbids `console.log` at the source level is
preferable: it catches the call before it ships rather than silently discarding
output at runtime.
