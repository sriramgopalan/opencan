# ADR-007: Webhook Delivery Model

**Status:** Accepted  
**Date:** 2026-06-23  
**Author:** sriramgopalan

---

## Context

Gap 6 adds outbound webhooks so admins can push post and comment events to external systems
(Slack, Linear, Discord, etc.) without first-party integrations. Two delivery models were
considered:

- **A — Synchronous best-effort**: sign and POST inside the originating request; retry once on
  timeout/network error; give up after that.
- **B — Redis queue (BullMQ)**: persist events to a queue; a separate worker delivers them with
  full retry semantics and at-least-once guarantees.

## Decision

**Option A — synchronous best-effort** for v1.

## Rationale

- The target event volume for an OpenCan instance is low (tens to hundreds of posts/day); a
  synchronous call with a 5-second timeout is acceptable latency on the mutation path.
- Option B requires a persistent worker process, BullMQ/pg queue infrastructure, and failure
  dashboards — significant operational complexity for a feature with no proven demand yet.
- Fire-and-forget from the tRPC mutation (`.catch(log)`) means the user-facing response is
  never delayed by webhook failures.
- If a webhook target is persistently slow or unreachable it is deactivated manually by the
  admin — there is no automated circuit-breaker in v1.

## Known trade-offs

| Trade-off | Impact |
|-----------|--------|
| Missed events if worker crashes mid-request | Low: events are not financial transactions |
| No delivery history / dead-letter queue | Admin must monitor via external logging |
| One slow webhook adds ~5 s of background work per event | Acceptable for v1 volume |

## Upgrade path (v2)

Replace `dispatchWebhook` internals with a BullMQ producer. The router call site stays
identical — the switch is fully internal to `lib/webhook.ts`.
