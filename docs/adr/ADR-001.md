# ADR-001: All API Access via tRPC Routers Only

## Status
Accepted

## Date
2026-06-13

## Context

Next.js App Router supports multiple data-access patterns simultaneously: React
Server Components can call repository functions directly (same process, no HTTP
hop), Client Components can call `fetch('/api/...')`, and route handlers can be
created ad hoc in `src/app/api/`. Without a firm rule, all three patterns
accumulate in parallel, and the codebase ends up with:

- Unvalidated `fetch` calls where the request and response shapes are asserted
  with `as SomeType` and never verified at runtime.
- Auth checks duplicated (or forgotten) across individual route handlers.
- No single place to find all the operations the application exposes to
  clients — some in `app/api/`, some in tRPC, some inlined in Server Actions.
- End-to-end type safety broken the moment any component uses raw `fetch`.

Etash's API surface is entirely owned by this codebase. There is no requirement
to expose a public REST API at this time. The full benefit of tRPC — a single
type-safe call graph, shared input validation, and centralised middleware — is
only realised if it is the *only* data path.

## Decision

All data mutations and queries initiated by client code must go through tRPC
procedures defined in `/server/routers/`.

**React Server Components** that need data call repository functions directly
(`/server/repositories/`) in the same Node.js process. They do not call tRPC
procedures internally and do not use `fetch`. This is not an exception to the
rule — RSCs are server code calling server code.

**Client Components** must use the tRPC React hooks (`api.board.create.useMutation()`,
`api.board.list.useQuery()`, etc.) exclusively. They must not:
- Call `fetch('/api/...')` directly.
- Import repository functions (which would bundle server code client-side).
- Use Next.js Server Actions as a parallel data layer.

**Route handlers** (`app/api/route.ts`) are permitted only for:
- Webhook receivers from third-party services (Stripe, GitHub, etc.) that
  cannot send tRPC-shaped payloads.
- OAuth callback endpoints.

Every route handler that is created must be documented with a comment
explaining why tRPC cannot satisfy the requirement.

## Consequences

**For development:**
- Any new data operation starts in `/server/routers/`, not with a `fetch` call.
  The router is the definition of the API.
- Input validation (Zod) and auth middleware (`protectedProcedure`) apply
  automatically to every procedure. There is no path for a client to reach
  the database that bypasses these layers.
- TypeScript catches client/server contract mismatches at compile time. Renaming
  a procedure field breaks the client import, not a runtime `JSON.parse`.
- All procedures are enumerable: the full API surface is the contents of
  `/server/routers/`.

**For testing:**
- tRPC caller factories (`createCallerFactory`) allow integration tests to call
  procedures directly without an HTTP server, with full type safety.
- There is no need to test route handlers for standard CRUD — only the webhook
  and OAuth exceptions need HTTP-level tests.

**Trade-offs:**
- Developers unfamiliar with tRPC face a learning curve. The tRPC docs and the
  patterns in `/server/routers/` serve as the reference.
- Webhook route handlers require boilerplate (raw body parsing, signature
  verification) that tRPC abstracts away for normal procedures. This is
  acceptable given the narrow scope of their use.

## Alternatives Considered

**Plain Next.js route handlers (`app/api/`) for all endpoints.**
Rejected. No shared input validation, no type-safe client, auth must be
checked manually in every handler. The risk of a missed auth check outweighs
the familiarity benefit.

**GraphQL (e.g. Pothos + Apollo).**
Rejected. Higher setup complexity, requires a schema definition language
alongside TypeScript types, and provides no meaningful advantage over tRPC for
an application with a single client and a single server. tRPC achieves the same
end-to-end type safety with less ceremony.

**Server Actions for mutations, tRPC for queries.**
Rejected. Splitting the data layer by operation type (mutations vs. queries)
means two sets of conventions, two places to look for auth checks, and two
testing patterns. Consistency is more valuable than the marginal ergonomic
benefit Server Actions offer for form submissions.

**Direct Prisma access from Client Components via Server Actions.**
Rejected. Collapses the boundary between data access and presentation, makes
the data layer untestable in isolation, and risks bundling Prisma client code
into the client bundle.
