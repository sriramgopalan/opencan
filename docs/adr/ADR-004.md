# ADR-004: Auth Checked in Middleware Only

## Status
Accepted

## Date
2026-06-13

## Context

Authentication and authorisation are the most common sources of security
regressions in web applications (OWASP A01: Broken Access Control). The failure
mode is almost always the same: a developer writes a new endpoint, forgets to
add the auth check, and ships an unauthenticated route to production.

In a tRPC codebase, the naive pattern is:

```ts
export const getBoard = t.procedure
  .input(z.object({ id: z.string() }))
  .query(async ({ input, ctx }) => {
    if (!ctx.session) throw new AppError('UNAUTHORIZED', 'Not logged in');
    // ... rest of the handler
  });
```

This pattern has two failure modes:

1. **Omission**: a developer adds a new procedure and forgets the `if (!ctx.session)` check. The procedure is now publicly accessible. There is no compiler or linter check that would catch this.

2. **Inconsistency**: different procedures check auth at different points (before vs. after input parsing, before vs. after the first DB call), with different error messages and codes. Auditing auth coverage requires reading every procedure body.

Etash's session is managed by a session library (e.g. NextAuth.js) that
populates `ctx.session` in the tRPC context factory. The session object is
available to middleware before any procedure body executes.

## Decision

Authentication is enforced by tRPC procedure type, not by code inside procedure
bodies.

Two base procedure types are defined in `/server/trpc.ts`:

```ts
export const publicProcedure = t.procedure;

export const protectedProcedure = t.procedure.use(authMiddleware);
```

`authMiddleware` reads `ctx.session`, throws `AppError('UNAUTHORIZED', ...)` if
no valid session exists, and narrows the context type so that `ctx.session.user`
is typed as non-nullable in all downstream procedures.

**Every procedure that requires authentication uses `protectedProcedure`.**
It never uses `publicProcedure` and then checks `ctx.session` inside the body.

**Authorisation** (verifying that the authenticated user may act on a specific
resource) is still checked inside procedure bodies, because it requires the
procedure's input:

```ts
export const deleteBoard = protectedProcedure
  .input(z.object({ id: z.string() }))
  .mutation(async ({ input, ctx }) => {
    const board = await boardRepo.getBoardById(input.id);
    if (!board) throw new AppError('NOT_FOUND', 'Board not found');
    if (board.ownerId !== ctx.session.user.id) throw new AppError('FORBIDDEN', 'Access denied');
    return boardRepo.deleteBoard(input.id);
  });
```

This ownership check cannot be in middleware because middleware does not have
access to the parsed input or the fetched resource. It is a required pattern in
every procedure that touches user-owned data, documented and enforced through
code review and the security skill file.

**Admin-only procedures** use a second middleware layer:

```ts
export const adminProcedure = protectedProcedure.use(adminMiddleware);
```

`adminMiddleware` checks `ctx.session.user.role === 'ADMIN'` and throws
`AppError('FORBIDDEN', 'Admin only')` if not satisfied.

No procedure checks `ctx.session.user.role` inside its body — role gates are
always a middleware concern.

## Consequences

**For development:**
- Adding a new authenticated procedure means typing `protectedProcedure` and
  nothing else. Auth is not a checklist item; it is structural.
- Auditing authentication coverage is a grep: `publicProcedure` returns every
  unauthenticated route. The list should be short and intentional (login,
  register, public board view for `isPublic: true` boards).
- The TypeScript type of `ctx.session.user` is non-nullable inside every
  `protectedProcedure` body. Accessing `ctx.session?.user?.id` with optional
  chaining inside a protected procedure is a type error — the compiler confirms
  the session is present.
- New role types (e.g. `MODERATOR`) become new middleware functions, not new
  in-body conditionals scattered across procedures.

**For testing:**
- Testing an unauthenticated request against a `protectedProcedure` is done by
  passing an unauthenticated context to the tRPC caller. The middleware throws
  before the procedure body runs — no procedure body code needs to be tested
  for the auth rejection case.
- Auth middleware is tested once, in isolation, covering the session-missing and
  session-expired cases.

**Trade-offs:**
- Developers must understand the `publicProcedure` / `protectedProcedure` /
  `adminProcedure` distinction and choose correctly. Choosing `publicProcedure`
  for a route that should be protected is still possible; code review and the
  grep audit are the safeguards.
- Authorisation (ownership checks) remains in procedure bodies and is not
  structurally enforced. This is an accepted limitation: ownership depends on
  runtime data that middleware cannot access. The security skill file and code
  review are the enforcement mechanisms.

## Alternatives Considered

**Per-procedure `if (!ctx.session)` checks.**
Rejected. This is the pattern that produces auth regressions. It requires every
developer to remember the check on every new procedure. One omission is a
security vulnerability.

**Route-level middleware in Next.js (`middleware.ts`).**
Rejected for tRPC procedures. Next.js middleware runs at the edge on the URL
path, before the request reaches the tRPC handler. It cannot access the parsed
procedure name or input, cannot enforce procedure-level auth granularity, and
cannot enrich the tRPC context. It is appropriate for redirect-based auth
(protecting entire route segments) but not for procedure-level access control.
Both can be used together — edge middleware for page-level redirects, tRPC
middleware for procedure-level auth.

**Decorator-based auth (`@Authenticated()` on procedure classes).**
Rejected. tRPC does not use class-based procedures. Decorator patterns require
a transformation layer (e.g. `reflect-metadata`) that adds complexity without
benefit over tRPC's native middleware chaining.

**Checking auth in the tRPC context factory (before any procedure runs).**
Rejected for the general case. The context factory cannot throw selectively —
it would reject all requests, including `publicProcedure` calls. Some procedures
(login, public board view) must be accessible without a session.
