# ADR-003: All Database Access Through Repository Functions

## Status
Accepted

## Date
2026-06-13

## Context

Prisma Client is importable anywhere in a Node.js codebase. Without a
structural rule, Prisma calls accumulate in tRPC routers, React Server
Components, Server Actions, route handlers, and utility functions — wherever a
developer first needs data. This produces:

- No single place to enforce query-level concerns: field selection, soft-delete
  filtering, tenant isolation, audit logging, or caching.
- Unit tests that cannot isolate business logic from the database. Testing a
  tRPC procedure requires either a real database or mocking Prisma's deeply
  nested method chain at the call site, which couples tests to implementation
  details.
- Queries that grow in complexity over time with no clear ownership. A
  `prisma.board.findMany({ where: { ... }, include: { ... } })` call inline in
  a router is hard to reuse, hard to paginate consistently, and hard to add an
  index hint to without touching every call site.
- Dependency-cruiser cannot enforce the architectural boundary if there is no
  boundary to enforce.

Etash's data layer will grow to include soft-delete, multi-tenancy scoping, and
row-level security. These cross-cutting concerns are only manageable if all
database access flows through a single, controlled layer.

## Decision

All Prisma Client calls must be made inside repository functions located in
`/server/repositories/`. No other file in the application may import or call
`prisma` directly.

**Repository structure:**
- One file per Prisma model: `user.ts`, `board.ts`, `feedbackResponse.ts`.
- Each file exports named functions describing intent: `getUserByEmail`,
  `createBoard`, `listBoardsByOwner`, `markFeedbackResolved`.
- No function name references Prisma methods (`prismaFindUnique`, `findManyWhere`).
- Cross-model operations (e.g. a transaction that writes to two models) are
  composed at the router or service layer by calling multiple repository
  functions within a `prisma.$transaction(...)`, not by writing a repository
  function that imports from another repository file.

**Enforcement:**
Dependency-cruiser rule `no-raw-prisma-outside-repositories` is configured in
`.dependency-cruiser.cjs` and runs in CI. Any import of `@prisma/client` or
`@/server/db` from outside `/server/repositories/` fails the build.

**The Prisma client singleton** lives in `/server/db.ts` and is imported only
by repository files.

## Consequences

**For development:**
- Adding a cross-cutting concern (e.g. filtering soft-deleted rows) requires
  a change in one place per model, not a search across the codebase.
- Query intent is named: `listBoardsByOwner({ ownerId, cursor, limit })` is
  easier to read in a router than the equivalent `prisma.board.findMany({
  where: { ownerId, deletedAt: null }, take: limit, ... })`.
- Repositories are the natural home for query-level performance decisions:
  which fields to select, which indexes to use, when to batch. These decisions
  are isolated from business logic.
- Repository functions are the boundary for unit test mocking. Tests for tRPC
  routers mock at the repository import, not at the Prisma method level. This
  means router tests are not coupled to the exact Prisma API used internally.

**For testing:**
- Repository unit tests mock `prisma` (via `prismaMock` in `/tests/`) and
  verify that the correct Prisma call was made with the correct arguments.
- Router integration tests call real repositories against the test database.
  They do not mock repositories — the real data layer runs.
- The Prisma mock (`vitest-mock-extended` or equivalent) is configured once in
  `/tests/prismaMock.ts` and imported wherever needed.

**Trade-offs:**
- Small CRUD operations require two files (repository + router) where a
  developer might feel one would suffice. This overhead is intentional: the
  repository is the unit boundary for testing and the enforcement point for
  cross-cutting concerns.
- Repository functions must be written before they can be called. There is no
  shortcut of inlining a quick `prisma.x.findUnique` in a router for a
  one-off query.

## Alternatives Considered

**Active Record pattern (model classes with built-in query methods).**
Rejected. Active Record collapses the distinction between domain objects and
database access, makes mocking difficult, and does not fit Prisma's data-mapper
design. Prisma's generated client is already the data mapper; the repository
layer sits above it.

**Service layer that owns both business logic and queries.**
Rejected. A service layer that calls Prisma directly has the same testability
problem as inline Prisma calls. Separating repositories from services preserves
the ability to unit-test services with mocked repositories and unit-test
repositories with a mocked Prisma client independently.

**Direct Prisma calls allowed in React Server Components.**
Rejected. RSCs run on the server and could technically import Prisma safely,
but allowing this creates two authoritative data access layers (RSC + repository)
and means cross-cutting concerns must be applied in both places. All server-side
data access routes through repositories; RSCs call repositories, not Prisma.

**ORM-agnostic query builder (Drizzle, Kysely).**
Rejected at this stage. Prisma's generated client provides type-safe queries and
automatic migration generation that significantly reduce boilerplate for a small
team. The repository layer provides enough abstraction that swapping the
underlying ORM in the future does not require changes outside `/server/repositories/`.
