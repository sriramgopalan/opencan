# ADR-005: All Experimental Features Behind Flags in /lib/flags

## Status
Accepted

## Date
2026-06-13

## Context

Etash will offer a managed hosting tier alongside a self-hosted build. The two
deployment targets have different feature sets, and within the managed tier,
new features will be rolled out gradually rather than shipped to all users
simultaneously. Without a coordinated mechanism for this, the options are:

- **Branch-based release**: long-lived feature branches that accumulate merge
  debt and make it difficult to ship bugfixes independently of feature work.
- **Ad-hoc environment variables**: `if (process.env.ENABLE_X === 'true')` checks
  scattered across the codebase with no central inventory, no type safety, and
  no consistent evaluation pattern.
- **Database flags per user**: flexible but requires a DB read on every request
  that checks a flag, and conflates feature gating with user preferences.

The ad-hoc environment variable approach is the path of least resistance and the
one most likely to be adopted without a rule. It creates a different problem:
flags accumulate, are never cleaned up, and become load-bearing configuration
that no one is willing to remove because the blast radius is unknown.

Etash needs a flag system that is: typed, centrally inventoried, evaluatable
server-side without a DB round-trip per request, and structured to make
flag removal straightforward once a feature is fully shipped.

## Decision

All experimental features, incomplete features, and deployment-target variations
are gated behind feature flags defined in `/lib/flags.ts`.

**Flag definition:**

```ts
// /lib/flags.ts
export const flags = {
  aiSummaries: {
    description: 'AI-generated summaries on feedback boards',
    defaultValue: false,
    owner: 'product',
    trackingIssue: 'https://github.com/etash/etash/issues/142',
  },
  selfHostedBuild: {
    description: 'Disables managed-tier features in the self-hosted build',
    defaultValue: false,
    owner: 'platform',
    trackingIssue: null,
  },
} satisfies Record<string, FlagDefinition>;

export type FlagName = keyof typeof flags;
```

**Flag evaluation:**

```ts
// /lib/flags.ts
export async function isEnabled(flag: FlagName, context?: FlagContext): Promise<boolean> {
  // Evaluation order: env override → user-level override (from DB/cache) → default
}
```

Flags are evaluated **server-side only**. Client components receive the
evaluated boolean (e.g. as a prop from a Server Component or via a tRPC query)
— they never import `/lib/flags.ts` or evaluate flags themselves. This prevents
flag definitions and logic from being bundled into the client.

**Flag lifecycle:**

Every flag in `/lib/flags.ts` must have a `trackingIssue` link (or `null` for
permanent infrastructure flags like `selfHostedBuild`). When a feature is fully
shipped, its flag is removed from the file and all gating code is deleted in the
same PR. Flags are not left in place as "just in case" rollback switches after
a feature has been stable for one release cycle.

**What requires a flag:**
- Any feature not ready to ship to 100% of managed-tier users.
- Any behaviour difference between the self-hosted and managed builds.
- Any integration with a third-party service that may not be configured in all
  environments (e.g. an AI provider that requires an API key).

**What does not require a flag:**
- Bug fixes.
- Refactors with no user-visible behaviour change.
- New UI that replaces existing UI in the same PR (not a gradual rollout).

## Consequences

**For development:**
- The full inventory of in-progress and environment-specific features is visible
  in one file. Reviewing `/lib/flags.ts` answers "what is currently behind a flag?"
- TypeScript catches references to non-existent flags at compile time
  (`isEnabled('typoedFlagName')` is a type error).
- Removing a flag is a grep for `isEnabled('flagName')` followed by deleting the
  gating code. The tracking issue serves as the reminder to do this.
- Third-party integrations that are not universally configured (AI providers,
  analytics) are always gated, preventing runtime errors in environments where
  the integration is not set up.

**For the self-hosted build:**
- The `selfHostedBuild` flag disables managed-tier features (usage-based billing,
  managed database backups, white-glove onboarding) in a single evaluation point
  rather than via scattered `if (process.env.MANAGED === 'true')` checks.

**Trade-offs:**
- Every experimental feature requires a flag definition before the first line of
  feature code. This is a small overhead that pays for itself by making the
  feature safely shippable before it is complete.
- Flag evaluation adds a layer of indirection that can make code paths harder to
  follow during development. The `trackingIssue` field is the reminder that this
  indirection is temporary.

## Alternatives Considered

**Ad-hoc `process.env.ENABLE_X` checks.**
Rejected. No central inventory, no type safety, inconsistent evaluation
(some checks are `=== 'true'`, some are `!== undefined`), and flags
accumulate indefinitely because there is no structured removal process.

**Database-persisted flags per user (à la LaunchDarkly).**
Rejected as the primary mechanism. A DB read on every flag evaluation adds
latency to every request and creates a hard dependency on the database for
feature gating. Useful for gradual percentage rollouts at scale; premature at
Etash's current stage. The flag system is designed to be replaceable with a
service like this later if needed — the `isEnabled` function is the single
integration point.

**Long-lived feature branches.**
Rejected. Feature branches that last more than a few days accumulate merge
conflicts, cannot receive hotfixes cleanly, and delay integration testing.
Trunk-based development with feature flags is the industry-standard alternative.

**Environment-specific Next.js configuration (`next.config.js` env blocks).**
Rejected. Next.js env configuration is evaluated at build time, not at request
time. It cannot support per-user or per-tenant flag variation, and it cannot be
changed without a redeploy. It is appropriate for truly static build-time
configuration (public URL, analytics write keys) but not for feature gating.
